# 森林老师个人站内容升级设计规格

**日期：** 2026-06-17  
**项目：** `D:\kaifa\senlin_website`  
**目标：** 在保留现有首页科技感与信息主线的前提下，新增更丰富的内容展示层，让访问者更立体地了解赵森林老师的教学内容、抖音输出、应用作品和真实教学现场，并保持源码可维护、后续可持续扩展。

---

## 1. 背景与设计结论

现有本地站点已经基本复刻了线上页面，但核心问题有三类：

1. 信息结构偏“简历式”，缺少能快速建立信任感和真实感的动态内容。
2. 应用能力、教学成果、个人内容输出之间缺少统一的展示容器，访问者无法形成“这个人既能教、也能做、还在持续输出”的完整印象。
3. 现有 `src/App.jsx` 与 `src/data.js` 存在明显中文乱码，继续增量修改会降低可维护性。

本次升级的最终设计结论如下：

1. 保留现有首页 Hero、数据、能力、荣誉、联系方式等主体框架。
2. 在中段新增一个独立的“内容宇宙”展示区域，拆分为三个子模块：
   - `精选视频`
   - `应用矩阵`
   - `教学现场环形相册`
3. 所有对外链接统一在新标签页打开，用户无需离开首页上下文。
4. 新增内容必须改成数据驱动，后续可通过维护数据数组继续扩展。
5. 在实现新增功能时，同步修复现有乱码文本问题，避免后续维护成本持续上升。

---

## 2. 设计目标

本次升级要同时满足以下目标：

1. 让用户在 30 秒内理解赵森林老师的身份不是单一“教师介绍”，而是“内容创作者 + 应用设计者 + 编程教育实践者 + 带队导师”。
2. 让站内新增内容不仅“能看”，还要“能点进去继续了解”，因此视频和应用都必须直接外跳。
3. 让照片不再只是静态堆叠，而是形成具有记忆点的沉浸式视觉模块。
4. 保持当前站点的视觉语言延续，避免新增区域像另一个模板拼接上来。
5. 确保项目继续是可维护源码项目，而不是一次性静态拼装页。

## 5. 新增模块一：精选视频

### 5.1 模块目标

通过两张精选视频卡，建立“赵森林老师在持续做内容输出”的认知，并引导用户跳转到抖音继续了解。

### 5.2 首批内容

首批固定接入两条用户提供的视频：

1. `交互式知识理解工具用起来`
   - 链接：`https://v.douyin.com/t9L2hN3pYNA/`
   - 定位：AI 工具 / 知识理解 / 学习方法
2. `免费英语资源赶紧用起来`
   - 链接：`https://v.douyin.com/nYzhLyIvGDc/`
   - 定位：学习资源 / 英语启蒙 / 工具推荐

### 5.3 展示形式

每个视频卡展示以下信息：

1. 标题
2. 主题标签
3. 一句简短说明
4. 外跳按钮，文案为 `打开抖音观看`

### 5.4 交互规则

1. 卡片整体可 hover 上浮并出现发光边框。
2. 点击按钮时，新标签页打开抖音链接。
3. 移动端保持完整点击区域，按钮不依赖 hover 才可见。

### 5.5 不做的内容

本阶段不实现：

1. 抖音 iframe 嵌入
2. 视频封面实时抓取
3. 站内视频播放弹层

原因是这些方案依赖外部平台嵌入稳定性，且不利于把用户引导至抖音主页进一步浏览。

## 6. 新增模块二：应用矩阵

### 6.1 模块目标

把“教什么”和“做了什么”从文字介绍升级为可点击体验的应用矩阵，让访问者直接感知作品落地能力。

### 6.2 展示策略

应用矩阵分为一个主卡和多个次级卡：

1. 主卡：`Python冒险岛`
2. 次级卡：
   - `享打字`
   - `AI Research`
   - `Code Research`
   - `AI互动课堂`
   - `模型训练`
   - `仿真模拟`

### 6.3 卡片信息

每个卡片至少包含：

1. 应用名称
2. 应用类型
3. 简短说明
4. 外跳动作提示，例如 `立即体验`

### 6.4 链接策略

1. 所有卡片点击后直接打开最终应用页。
2. 不先跳转到站内详情页。
3. 统一 `target="_blank"` + `rel="noreferrer"`。

### 6.5 数据来源与目标域名方向

根据用户提供信息和已有文档提取结果，本期应用入口优先围绕以下站点整理：

1. `class.codebn.cn`
2. `game.codebn.cn`
3. `ai.codebn.cn`

其中：

1. `Python冒险岛` 作为最核心的游戏化编程入口，优先放主视觉卡。
2. `享打字`、`AI Research`、`Code Research` 作为用户明确要求展示的功能入口。
3. `AI互动课堂`、`模型训练`、`仿真模拟` 对应教学软件、Teachable Machine、PhET 等方向。

对无法从现有资料中精确恢复最终 URL 的条目，实施阶段统一接入对应站点的稳定首页或功能总入口，并将 URL 独立配置在数据文件中，后续只需修改配置即可替换目标地址；不得将卡片链接留空。

### 6.6 视觉层级

1. 主卡在布局、尺寸、光效、文案权重上明显更强。
2. 次级卡使用统一矩阵布局，保持整洁和可扩展性。
3. 视觉上强调“应用实验室 / 项目矩阵”的感觉，而不是常规博客卡片列表。

## 7. 新增模块三：教学现场环形相册

### 7.1 模块目标

通过一个可旋转、可暂停、可放大的 3D 环形相册，强化“真实带学生、真实上课、真实比赛、真实成果”的沉浸式感受。

### 7.2 动效结论

采用 3D 圆环式照片布局，而非平面轮播或瀑布流。原因如下：

1. 记忆点更强，符合用户“炫酷”“圆环形”“卡片旋转”的明确要求。
2. 能在不占用过多垂直空间的情况下展示更多内容。
3. 与现有站点的科技感背景和动效风格更一致。

### 7.3 交互规则

1. 相册默认自动缓慢旋转。
2. 鼠标悬停到相册区域时，整体旋转暂停。
3. 鼠标悬停到某张卡片时，该卡片抬升、亮边、显示分类标签。
4. 点击卡片后弹出图片放大预览层。
5. 预览层支持：
   - 点击遮罩关闭
   - 按 `Esc` 关闭
   - 显示图片标题或分类
6. 移动端不强制保留 3D 环形拖拽体验，可降级为横向滑动卡片流，但仍保留点击放大。

### 7.4 素材分类结论

根据目前已核验的图片内容，照片可分为以下真实分类：

1. `师生合照`
   - 课堂内一对一或一对多合影
   - 获奖后与学生单独合影
2. `学生团队`
   - 多名学生在教室、窗边、日常活动中的团队照片
3. `教学现场`
   - 指导学生操作电脑
   - 课堂展示或作品讲解
4. `比赛成果`
   - WRCC 奖台、赛事舞台、奖杯和证书展示
5. `活动出行`
   - 长城等非课堂但能体现陪伴与成长的活动场景

### 7.5 当前已确认的代表性素材方向

已确认可优先放入相册的内容类型包括：

1. 师生课堂合影
2. 学生团队合影
3. WRCC 领奖台与比赛背景照
4. 比赛现场调试与竞赛过程
5. 成果展示与证书合影
6. 活动场景自拍与出行照片

### 7.6 视觉要求

1. 每张卡片需有统一圆角、边框、玻璃光感或霓虹描边，保持整体精致感。
2. 不按文件名原样暴露给用户，展示时统一使用人工可读标签。
3. 放大预览要明显高于二维码弹层层级，避免遮罩冲突。

## 9. 技术设计边界

### 9.1 数据组织

新增内容必须采用数据驱动结构，至少拆分出以下数据集合：

1. 视频数据
2. 应用矩阵数据
3. 相册数据
4. 导航数据更新

这样后续新增内容时，不需要反复改复杂 JSX，只需维护数据配置。

### 9.2 组件边界

允许在实现阶段把 `App.jsx` 中新增的复杂区域拆成小组件，重点关注以下边界：

1. 视频卡组件
2. 应用卡组件
3. 环形相册组件
4. 图片预览弹层组件

如果 `App.jsx` 继续承担全部逻辑，会进一步变成难维护的大文件，因此允许针对新增复杂功能做适度拆分。

### 9.3 响应式策略

1. 桌面端优先展现完整 3D 感和层次感。
2. 平板端保留主要布局和动效，但减轻透视强度。
3. 手机端优先保证可读性和可点击性，允许把环形相册降级为横向滑动列表。

### 9.4 性能策略

1. 不引入额外重型依赖。
2. 使用现有 `framer-motion` 完成主要动效。
3. 图片展示应避免一次性触发过多复杂动画，控制卡片数量和动画幅度。

## 10. 错误处理与容错

### 10.1 外链容错

1. 外链点击应使用标准 `<a>` 标签而非复杂脚本跳转。
2. 即使目标站点临时不可用，也不应影响页面自身渲染。

### 10.2 图片容错

1. 相册数据应全部指向本地可访问图片路径。
2. 如果单张图片加载失败，不应导致整个相册崩溃。

### 10.3 交互容错

1. 图片弹层和现有二维码弹层不得互相污染状态。
2. 键盘关闭和遮罩关闭行为必须稳定。
